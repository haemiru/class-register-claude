# Plan: 강의신청 페이지 (class-register)

> PDCA Phase: **Plan** · Created: 2026-06-20 · Feature: `class-register`
> Stack: Vite + React + Supabase + Vercel · PG: 토스페이먼츠

---

## Executive Summary

| 관점 | 내용 |
|------|------|
| **Problem (문제)** | 강의를 개설할 때 신청 접수·정원 관리·참가비 수금을 카톡/수기/계좌이체로 처리해 누락·초과·정산 오류가 발생한다. |
| **Solution (해결)** | 관리자가 강의(날짜·장소·주제·정원·참가비)를 등록하면, 참여자가 강의를 선택해 카드결제까지 한 흐름으로 신청을 완료하는 웹 페이지. |
| **Function/UX Effect (기능·UX 효과)** | 관리자: 비밀번호 로그인 후 강의 CRUD + 신청자 목록 확인. 참여자: 강의 목록 → 신청서(이름·연락처) → 토스 카드결제 → 신청 확정 화면. 정원 마감 시 신청 자동 차단. |
| **Core Value (핵심 가치)** | "결제 = 신청 확정"으로 정원·수금을 자동 동기화해, 운영자 1명이 수작업 없이 강의 모집을 운영할 수 있다. |

## Context Anchor

| 항목 | 내용 |
|------|------|
| **WHY** | 강의 모집의 신청·정원·결제를 한 흐름으로 자동화해 수작업 오류와 정산 부담을 없앤다. |
| **WHO** | 관리자(강의 개설자 1명) · 참여자(수강 신청자, 비로그인 일반 사용자) |
| **RISK** | 결제는 됐는데 신청이 미확정되는 정합성 깨짐 / 동시 신청으로 인한 정원 초과 / PG 키·관리자 비밀번호 노출 |
| **SUCCESS** | 결제 성공 건은 100% 신청 확정으로 기록되고, 정원 초과 신청이 0건이다. |
| **SCOPE** | IN: 강의 CRUD, 신청 흐름, 토스 카드결제, 정원 마감 차단, 신청자 목록. OUT(차후): 이메일/알림톡, 환불 자동화, 다중 관리자, 쿠폰·할인. |

## 1. Background & Problem

- 피지오 후각 연구소 / 짱샘에듀 강의를 모집할 때 신청을 카톡·구글폼·계좌이체로 받으면:
  - 정원 초과/중복 신청을 사람이 일일이 확인해야 함
  - 참가비 입금 확인과 신청 확정이 분리돼 누락 발생
  - 신청자 명단을 따로 정리해야 함
- 기존 `regist-form`(코칭 예약)과 유사하나, **다회차 코칭이 아닌 단발성 강의 모집**에 맞춘 더 단순한 모델이 필요.

## 2. Goals (목표)

- G1. 관리자가 코드 없이 강의를 등록/수정/마감할 수 있다.
- G2. 참여자가 강의 선택 → 신청 → 카드결제를 끊김 없이 완료한다.
- G3. 결제 완료와 신청 확정이 원자적으로 연동된다(정합성).
- G4. 정원이 차면 추가 신청이 차단된다.

### Non-Goals (이번에 안 함)

- 이메일/카카오 알림톡 발송
- 자동 환불/취소 처리 (관리자가 토스 콘솔에서 수동 처리)
- 다중 관리자·권한 분리
- 쿠폰/할인/대기자 명단

## 3. Requirements

### 3.1 관리자 (Admin)

| ID | 요구사항 |
|----|----------|
| AR-1 | 단일 비밀번호로 관리자 페이지에 로그인한다 (환경변수 기반). |
| AR-2 | 강의 등록: 제목/주제, 날짜·시간, 장소, 정원, 참가비, 설명을 입력한다. |
| AR-3 | 강의 목록 조회·수정·마감(비공개) 처리. |
| AR-4 | 강의별 신청자 목록(이름·연락처·결제상태·신청시각)을 확인한다. |

### 3.2 참여자 (Participant)

| ID | 요구사항 |
|----|----------|
| PR-1 | 공개된 강의 목록을 본다 (정원/마감 상태 표시). |
| PR-2 | 강의를 선택하고 신청서(이름, 연락처 — 필수)를 작성한다. |
| PR-3 | 토스페이먼츠 카드결제 창으로 참가비를 결제한다. |
| PR-4 | 결제 성공 시 신청 내역 확인 화면을 본다. |
| PR-5 | 정원이 마감된 강의는 신청 버튼이 비활성화된다. |

### 3.3 결제/정합성 (Payment)

| ID | 요구사항 |
|----|----------|
| YR-1 | 토스 결제창 호출 → 결제 승인(서버 검증) → 신청 확정의 순서를 따른다. |
| YR-2 | 결제 승인은 서버(Serverless Function)에서 시크릿 키로 검증한다. 클라이언트 신뢰 금지. |
| YR-3 | 정원 카운트는 결제 승인 시점에 증가하며, 동시성 환경에서 초과되지 않는다. |

## 4. Success Criteria (검증 가능한 완료 기준)

- [ ] SC-1: 관리자가 비밀번호로 로그인 후 강의를 등록하면 목록에 노출된다.
- [ ] SC-2: 참여자가 강의 선택 → 신청서 → 토스 결제 → 확인 화면까지 도달한다.
- [ ] SC-3: 결제 승인이 서버에서 검증되고, 승인된 건만 신청 확정(`paid`)으로 저장된다.
- [ ] SC-4: 정원 N명이 모두 결제 완료되면 해당 강의 신청 버튼이 비활성화된다.
- [ ] SC-5: 관리자가 강의별 신청자 목록(이름·연락처·결제상태)을 확인할 수 있다.
- [ ] SC-6: 토스 시크릿 키·관리자 비밀번호가 클라이언트 번들에 노출되지 않는다(.env 서버 전용).

## 5. Data Model (초안, Supabase · 접두사 `cr_`)

```
cr_classes (강의)
  id (uuid, pk)
  title              text        -- 강의 주제/제목
  description        text        -- 설명
  location           text        -- 장소
  starts_at          timestamptz -- 날짜·시간
  capacity           int         -- 정원
  fee                int         -- 참가비(원)
  status             text        -- 'open' | 'closed'
  created_at         timestamptz default now()

cr_registrations (신청)
  id (uuid, pk)
  class_id           uuid  fk -> cr_classes.id
  name               text  not null   -- 신청자 이름
  phone              text  not null   -- 연락처
  payment_status     text  -- 'pending' | 'paid' | 'failed'
  toss_payment_key   text  -- 토스 결제 키
  toss_order_id      text  -- 주문 ID
  amount             int   -- 결제 금액
  created_at         timestamptz default now()
```

- 정원 체크: `paid` 상태 registration 수 < `capacity` 일 때만 신청 허용.
- 동시성: 결제 승인 서버 함수에서 카운트 확인 후 insert (또는 DB 제약/트랜잭션으로 보강 — Design에서 확정).

## 6. User Flows (요약)

**참여자**: 강의 목록 → 강의 상세/신청 → 신청서 입력 → 토스 결제창 → (성공) `/success` 결제 승인 검증 → 신청 확정 화면 / (실패) `/fail`

**관리자**: `/admin` 비밀번호 입력 → 강의 등록 폼 → 강의 목록(수정·마감) → 강의별 신청자 목록

## 7. Risks & Mitigations

| 리스크 | 영향 | 완화책 |
|--------|------|--------|
| 결제됐는데 신청 미확정 (정합성) | 높음 | 토스 successUrl → 서버 결제 승인 검증 후에만 registration을 `paid`로 확정. 승인 실패 시 결제 취소 안내. |
| 동시 신청으로 정원 초과 | 중간 | 승인 서버 함수에서 현재 paid 수 재확인 후 insert. 필요 시 DB 제약 추가. |
| 시크릿 키/비밀번호 노출 | 높음 | 시크릿 키·관리자 비번은 Serverless 환경변수만 사용, 클라이언트 번들 금지. (메모리: API 키 보안 원칙) |
| PG 계약/키 지연 | 중간 | 토스 테스트 키로 전체 흐름 선구현, 실 키는 발급 후 환경변수 교체. |

## 8. Open Questions (Design 단계에서 확정)

- 결제 승인 서버 로직을 Vercel Serverless Function으로 둘지, Supabase Edge Function으로 둘지.
- 정원 동시성 보장 방식(서버 재확인 vs DB 제약/트랜잭션).
- 관리자 페이지 라우트 보호 방식(비밀번호 세션 처리).

## 9. Milestones (개략)

1. M1 — 프로젝트 셋업 (Vite+React+Tailwind, Supabase 연결, 스키마)
2. M2 — 관리자: 비번 로그인 + 강의 CRUD + 신청자 목록
3. M3 — 참여자: 강의 목록 + 신청서
4. M4 — 토스 결제 연동 + 서버 승인 검증 + 정합성/정원 차단
5. M5 — 결제 확인 화면, 배포(Vercel)
